### 全可定制化 Web 聊天原型系统设计文档 (纯前端 PoC 版)

#### 1\. 项目概述与设计原则

-   **项目目标**：快速构建一个基于纯前端架构、直接与大模型交互的全可定制化聊天 App 原型（PoC）。
-   **核心特性**：

    1.  **无后端直连**：前端直接管理 API Key 并与大语言模型（LLM）进行对话交互。
    2.  **全可定制化视图**：聊天窗口通过 iframe 沙盒隔离，支持注入并渲染任意 HTML/JS/CSS 实现界面重绘。
    3.  **自适应多端布局**：一套代码适配移动端原生体验与桌面端的分屏视觉展示。
-   **设计原则**：

    -   **极简务实**：抛弃复杂的鉴权与云端数据库，数据全面本地化。
    -   **防御性编程**：充分考虑高度定制化带来的白屏风险，设计完善的 UI 重置与恢复策略。

#### 2\. 核心技术栈选型

-   **核心框架**：React，使用 Capacitor 打包为 iOS 原生应用，同时支持 Web 静态托管部署。
-   **本地存储**：IndexedDB 或 LocalStorage。用于持久化存储 API Key、Agent 设定、会话列表、聊天记录以及自定义界面的代码。
-   **沙盒容器**：HTML `iframe` 标签。负责隔离第三方或用户自定义的代码逻辑，保障主应用的稳定性与安全性。
-   **网络通信**：浏览器原生的 Fetch API（或对应大模型服务商的 NPM 包），用于直接向大模型服务商（如 OpenAI, Anthropic, DeepSeek, Moonshot, Gemini 等）发起流式或非流式请求。

#### 3\. UI 布局与跨端视觉架构

为了保证多端的体验统一且不空洞，采用**响应式折叠架构**。

-   **移动端 (iOS/H5) 视图**：

    -   采用经典的类微信底部导航栏（Bottom TabBar）设计。
    -   包含三大核心模块：**聊天**（最近会话列表）、**通讯录**（包含所有预设或创建的 Agent）、**设置**（用于配置全局 API Key 和应用偏好）。
-   **桌面/大屏端 (Web/Pad) 视图**：

    -   **左侧展示区**：固定展示品牌标识（Brand Logo）、应用名称或简单的系统状态。
    -   **中轴核心区**：严格保持移动端原生比例的竖屏容器，直接复用移动端的 UI 逻辑与交互，让用户在使用桌面版时依然有熟悉的手机端体验。
    -   **右侧填充区**：使用常规的视觉元素进行填充以避免空洞。例如放置品牌插画、当前对话 Agent 的详细设定面板，或者是快捷帮助文档。

#### 4\. 本地数据流与状态管理

所有数据在客户端本地闭环管理。

-   **Config Store**：存储用户的 LLM API Key、默认模型选择、全局系统配置。
-   **Agents Store (通讯录)**：存储每个 Agent 的定义，包括 Agent Name、System Prompt、头像、预设的 UI 风格或自定义 HTML 模板。
-   **Chats Store (会话列表)**：记录用户与某个 Agent 的聊天元数据，包括最后一条消息摘要、未读状态、以及当前会话是否启用了独立定制的 UI 代码。
-   **Messages Store**：以会话 ID 为维度的本地聊天记录集。

#### 5\. 核心业务工作流

**5.1 沙盒渲染与容灾防错流**

-   **渲染机制**：当用户在聊天列表点击某个会话进入聊天详情页时，主应用读取该会话绑定的自定义 HTML/JS/CSS 代码，并将其作为 Blob URL 或直接注入到 iframe 中渲染。
-   **边缘返回保障**：聊天详情页本身属于主应用的最顶层视图，必须强制保留 iOS 原生的侧滑返回手势（Swipe Back）以及左上角固定的物理/虚拟返回按钮。无论 iframe 内部如何渲染，返回外层列表的控制权始终在主应用手中。
-   **白屏重置机制**：若用户或 Agent 下发的自定义 UI 存在 Bug 导致 iframe 完全白屏或卡死，用户可以返回上一级的“聊天列表”页面。在此页面，用户可以通过**侧滑列表项（Swipe Actions）**或**长按会话卡片**呼出操作菜单，选择“恢复默认 UI”。主应用将清除该会话的自定义配置，下次进入时使用内置的标准聊天界面。

**5.2 纯前端大模型对话流**

1.  **消息输入**：用户在 iframe 内的输入框（或主应用提供的标准输入框）输入文本并发送。
2.  **内外通信**：如果是 iframe 内触发的发送，iframe 通过 `window.postMessage` 将用户文本传递给主应用（React 容器）。
3.  **请求组装与发送**：主应用拦截到发送动作，从本地取出 API Key 与该 Agent 的历史聊天上下文，组装成 LLM API 所需的格式，直接发起请求。
4.  **响应回传**：主应用拿到大模型的回复数据后，存入本地 Messages Store，并通过 `postMessage` 将新消息派发给 iframe，由自定义代码决定如何呈现这条回复。

  - 模板可以组合安装
